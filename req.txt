# 📄 DOCUMENTO DE ESPECIFICACIÓN DE REQUERIMIENTOS

## Sistema de Gestión de Tickets de Devoluciones y Reclamos

**Cliente:** Operador de Turismo
**Integración BackOffice:** Oris

---

# 1. INTRODUCCIÓN

## 1.1 Propósito

Definir los requerimientos funcionales, no funcionales, técnicos y de integración para el desarrollo de una aplicación web que permita:

* Ingresar solicitudes de reclamos, devoluciones u otras.
* Generar número correlativo de ticket.
* Extraer información automáticamente desde el sistema BackOffice **Oris**.
* Gestionar flujo de autorización.
* Asociar solicitudes a reservas y proveedores.
* Generar estadísticas y reportes.
* Administrar usuarios y configuraciones visuales.

---

# 2. ALCANCE DEL SISTEMA

La aplicación permitirá:

* Registrar tickets asociados a una reserva (File).
* Extraer automáticamente datos desde base de datos **Oris1**.
* Administrar tablas parametrizables.
* Gestionar flujo de aprobación.
* Generar estadísticas gráficas.
* Manejar identidad visual configurable.
* Integrarse mediante backend seguro a SQL Server.

---

# 3. ARQUITECTURA GENERAL

## 3.1 Arquitectura Propuesta

Arquitectura de 3 capas:

1. **Frontend Web**

   * Interfaz moderna (React / Angular / Vue).
   * Responsive.
   * Configurable en colores y logo.

2. **Backend API REST**

   * .NET Core / Node.js
   * Autenticación segura (JWT).
   * Conexión segura a SQL Server.

3. **Base de Datos**

   * SQL Server
   * Nueva base: `gclientes`
   * Integración lectura con base `Oris1`

# 4. REQUERIMIENTOS FUNCIONALES

---

## 4.1 Generación de Ticket

RF01: Cada registro ingresado debe generar automáticamente un número correlativo único de Ticket.
RF02: El número de Ticket será el identificador principal del sistema.
RF03: El correlativo debe ser incremental y no reutilizable.

Formato sugerido:
TKT-00000001

---

## 4.2 Registro de Solicitudes

RF04: El sistema debe permitir ingresar un nuevo registro seleccionando:

* Tipo de solicitud:

  * Reclamo
  * Devolución
  * Otras

RF05: Todas las tablas deben ser administrables (CRUD completo).

RF06: Debe permitir ingresar un texto descriptivo libre asociado a la solicitud.

RF07: Todo registro debe estar obligatoriamente asociado a un número de File (reserva).

RF08: Debe permitir asociar opcionalmente un proveedor vinculado a la reserva.

RF09: Debe permitir ingresar monto solicitado (si corresponde).

---

## 4.3 Integración con BackOffice Oris

Base de datos BackOffice: **Oris1**

### 4.3.1 Tablas involucradas

* Tabla reserva: `file_`

  * Campo identificador: `num_file`
  * agencia
  * f_viaje
  * ciudad
  * ope

* Tabla detalle servicios: `de_file`

  * Campo proveedor: `provee`

* Vista proveedores:

  * `view_hotel_operad`
  * Campo relación: `codigo`
  * Campo nombre proveedor

* Tabla operador:

  * `opecsts`
  * Campo: `codigo`

---

### 4.3.2 Comportamiento al ingresar número de File

RF10: Al ingresar el número de File, el sistema debe:

1. Validar existencia en Oris1.file_
2. Mostrar automáticamente en pantalla:

* Agencia → file_.agencia
* Fecha de viaje → file_.f_viaje
* Destino → file_.ciudad

3. Obtener proveedor(es) asociados desde:

```
de_file.provee
JOIN view_hotel_operad.codigo
```

4. Obtener operador desde:

```
file_.ope
JOIN opects.codigo
```

RF11: Debe permitir seleccionar uno o más proveedores asociados a la reserva.

---

## 4.4 Administración de Usuarios

RF12: El sistema debe manejar login con autenticación segura.
RF13: Debe permitir roles:

* Administrador
* Supervisor
* Ejecutivo
* Usuario Autorizador

RF14: Debe permitir CRUD de usuarios.

RF15: Control de acceso basado en roles.

---

## 4.5 Flujo de Autorización

RF16: El sistema debe permitir enviar solicitudes a autorización.
RF17: Debe permitir aprobar o rechazar solicitudes.
RF18: Debe enviar notificación interna a los usuarios que deben autorizar.
RF19: Debe mantener historial de autorizaciones.
RF20: Debe permitir cierre de casos.

Estados del ticket:

* Nuevo
* En revisión
* Pendiente autorización
* Aprobado
* Rechazado
* Cerrado

---

## 4.6 Mensajería Interna

RF21: El sistema debe enviar notificaciones internas cuando:

* Se crea un ticket.
* Se requiere autorización.
* Se aprueba/rechaza.
* Se cierra caso.

RF22: Debe existir bandeja de mensajes por usuario.

---

## 4.7 Estadísticas y Reportes

RF23: El sistema debe generar gráficos dinámicos por:

* Proveedor
* Agencia
* Vendedor
* Monto devuelto
* Estado
* Tipo de solicitud

RF24: Dashboard general con indicadores:

* Total reclamos
* Total devoluciones
* Tiempo promedio resolución
* Monto total devuelto

RF25: Exportación a Excel y PDF.

---

## 4.8 Configuración Visual

RF26: Permitir carga de logo institucional.
RF27: Permitir configurar paleta de colores.
RF28: Configuración debe ser administrable desde módulo administración.
